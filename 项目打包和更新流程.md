## 二、首次打包流程

### 2.1 准备工作

1. **清理测试文件**（可选但推荐）：
   - 删除所有 `test_*.py` 文件
   - 清空 `screenshots/` 目录（保留目录结构）
   - 删除 `__pycache__/` 目录

2. **验证核心文件完整**：
   - 确保所有核心文件都存在
   - 检查 `requirements.txt` 是否完整

### 2.2 打包交付内容

交付给开发同学的压缩包应包含以下文件：

```

├── web_app_complete.py
├── friendly_link_monitor.py
├── templates/
│   └── index_complete.html
├── requirements.txt
├── Dockerfile
├── render.yaml
├── README.md
├── DEPLOY.md
└── 启动服务.command
```

### 2.3 打包步骤

```bash

# 2. 创建临时目录
```

## 三、版本控制建议

### 3.1 使用Git管理代码（推荐）

为了便于后续更新，建议使用Git进行版本控制：

```bash
# 初始化Git仓库
cd /Users/wangjiuchen/Documents/trae_projects/website-link-monitor-clean
git init

# 创建.gitignore文件
cat > .gitignore << 'EOF'
__pycache__/
*.pyc
*.pyo
*.pyd
.Python
*.so
*.egg
*.egg-info/
dist/
build/
screenshots/*.png
screenshots/*.jpg
.DS_Store
EOF

# 添加文件并提交
git add .
git commit -m "Initial commit - v1.0"

# （可选）推送到GitHub
# git remote add origin <your-repo-url>
# git push -u origin main
```

### 3.2 版本号管理

建议使用语义化版本号：`v主版本.次版本.修订号`

- **主版本**：重大功能改动或架构调整
- **次版本**：新增功能
- **修订号**：bug修复和小改动


## 七、常见问题

### Q1：每次修改都需要重新打包吗？

**A：** 不需要！建议使用Git进行版本控制，这样：
- 开发同学可以直接 `git pull` 获取更新
- 有完整的版本历史记录
- 可以轻松回滚到任意版本

### Q2：如何快速回滚到上一个版本？

**A：** 如果使用Git：
```bash
git log --oneline          # 查看提交历史
git checkout <commit-hash>  # 回滚到指定版本
```

### Q3：哪些文件不应该交付？

**A：** 以下文件不应包含在交付包中：
- `__pycache__/` 目录
- 所有 `test_*.py` 测试文件
- `screenshots/` 目录下的截图文件
- `.DS_Store` 等系统文件
- 本地配置文件（如果有的话）

### Q4：如何验证更新是否成功？

**A：** 部署后检查：
1. 服务正常启动
2. 访问页面正常
3. 测试核心功能（链接检测、截图等）
4. 检查日志无错误
